# 구현 태스크 (MVP)

아래 순서는 의존성 기준입니다. 병렬 가능한 항목은 같은 단계 내에서 분담할 수 있습니다.

## Phase 1 — 뼈대

- [ ] **T1** 저장소 구조 결정(모노레포 vs 단일 서비스) 및 `README`에 로컬 실행 절차 초안
- [ ] **T2** 로컬 웹 서버 스캐폴딩(API + 정적 또는 SPA 빌드 산출물 서빙)
- [ ] **T3** 환경 변수 스키마 정의(예: LLM 엔드포인트/모델명/API 키) 및 `.env.example` 제공

## Phase 2 — 문서 파이프라인

- [ ] **T4** `.txt` 업로드 → UTF-8 정규화 및 길이 제한/에러 처리
- [ ] **T5** `.pdf` 업로드 → 텍스트 추출(라이브러리 선택 및 실패 시 사용자 메시지)
- [ ] **T6** 추출 텍스트를 `document_id`와 함께 **로컬 저장소**(파일 또는 경량 DB)에 보관

## Phase 3 — 요약·리포트 (MVP 1순위)

- [ ] **T7** 추출 텍스트 → 구조화된 요약·리포트 생성(섹션 템플릿 고정)
- [ ] **T8** 결과를 API로 반환하고 UI에 온스크린 렌더링(마크다운 또는 단순 HTML)

## Phase 4 — 문서 근거 Q&A

- [ ] **T9** 질문 + 해당 문서 컨텍스트를 모델에 전달하는 프롬프트(“문서 밖 추측 금지” 규칙)
- [ ] **T10** 긴 문서 시 청크/검색 전략(단순 전체 주입 vs RAG) 중 하나를 선택하고 구현
- [ ] **T11** 응답에 짧은 근거 인용(선택) 또는 “문서에 없음” 처리

## Phase 5 — 세션·UX

- [ ] **T12** 로컬 세션 목록·선택·삭제(브라우저 저장소 또는 서버측 로컬 파일)
- [ ] **T13** 빈 상태·로딩·오류 토스트 등 최소 UX 마감
- [ ] **T14** 수동 스모크 시나리오 문서화(샘플 PDF/txt 포함)

## 완료 정의

- `docs/MVP_SPEC.md` 수용 기준 전부 충족
- `seed.yaml`의 `acceptance_criteria`와 불일치 없음
